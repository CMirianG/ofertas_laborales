CAPÍTULO II: MARCO TEÓRICO Y ARQUITECTURA DEL SISTEMA

2.1. FUNDAMENTOS TEÓRICOS

2.1.1. Aplicaciones Web y Arquitectura Cliente–Servidor

El sistema de ofertas laborales se desarrolla como una aplicación web basada en la arquitectura cliente–servidor, modelo ampliamente utilizado en sistemas de información modernos. En este modelo, el cliente corresponde al navegador web del usuario, que se encarga de presentar la interfaz gráfica, capturar las acciones del usuario y enviar solicitudes HTTP al servidor. El servidor, por su parte, procesa dichas solicitudes, ejecuta la lógica de negocio, accede a los datos almacenados y genera la respuesta que será visualizada en el cliente.

Esta separación lógica ofrece varias ventajas. En primer lugar, permite centralizar la lógica y los datos en un único punto (el servidor), lo que simplifica las tareas de mantenimiento, seguridad y actualización del sistema: cualquier mejora o corrección realizada en el servidor se refleja inmediatamente para todos los usuarios, sin necesidad de instalar nada en sus equipos. En segundo lugar, facilita la escalabilidad, ya que es posible ajustar los recursos del servidor (memoria, CPU, conexiones a base de datos) para atender a un mayor número de usuarios concurrentes sin alterar los clientes.

En el contexto de la Oficina de Responsabilidad Social Universitaria (RSU), esta arquitectura permite que diversos actores —personal de RSU, docentes, estudiantes, egresados— accedan al sistema desde distintos dispositivos (computadoras de escritorio, laptops, tablets) usando únicamente un navegador web, mejorando la accesibilidad y reduciendo barreras tecnológicas para la adopción de la herramienta.

2.1.2. Arquitectura por Capas y Patrón Modelo–Vista–Controlador (MVC)

Para gestionar la complejidad del sistema y facilitar su evolución, se adopta una arquitectura por capas inspirada en el patrón Modelo–Vista–Controlador (MVC). Este patrón propone dividir el software en componentes con responsabilidades claramente diferenciadas, evitando que la lógica de negocio se mezcle con la presentación o con los detalles de acceso a datos.

En el sistema de ofertas laborales, esta separación se materializa de la siguiente manera:

  • Capa de Modelo: Representa los datos y las reglas de negocio asociadas. En este proyecto, está implementada principalmente a través del servicio MongoDBManager, que define cómo se almacenan y recuperan las ofertas laborales, los usuarios y las estadísticas en la base de datos MongoDB. Esta capa concentra operaciones como consultas, inserciones, actualizaciones y agregaciones.
  • Capa de Vista: Corresponde a la presentación de la información al usuario. Se implementa mediante plantillas HTML empleando el motor de plantillas Jinja2, que permite generar contenido dinámico a partir de los datos suministrados por los controladores. En esta capa se cuida la organización de la información (tablas, gráficos, formularios) y la usabilidad de la interfaz.
  • Capa de Controlador: Actúa como intermediaria entre la vista y el modelo. Está constituida por los blueprints de Flask (auth.py, dashboard.py, ofertas.py), que reciben las peticiones HTTP, validan datos de entrada, coordinan llamadas a los servicios de negocio (scraping, acceso a datos) y deciden qué vista se debe renderizar.

Esta estructura promueve la alta cohesión interna de cada módulo (cada uno hace pocas cosas, pero bien definidas) y un bajo acoplamiento entre módulos (los cambios en una capa afectan mínimamente a las otras). Como consecuencia, se facilita la incorporación de nuevas funcionalidades, la corrección de errores y la futura incorporación de otros canales de acceso (por ejemplo, una API pública) sin reescribir el sistema desde cero.

2.1.3. Desarrollo Web con Microframeworks: Flask

Flask es un microframework para el desarrollo de aplicaciones web en Python que se caracteriza por su filosofía de ser “ligero pero extensible”. A diferencia de frameworks más monolíticos, Flask ofrece un conjunto básico de funcionalidades —gestión de peticiones y respuestas, sistema de rutas, manejo de cookies y sesiones, integración con plantillas— y permite que el desarrollador seleccione de manera explícita las extensiones adicionales que necesite (autenticación, formularios, ORM, etc.).

Esta cualidad se ajusta muy bien a las necesidades del Sistema de Ofertas Laborales para Tacna, ya que se requiere una solución que sea al mismo tiempo ágil de desarrollar y flexible para crecer. Con Flask es posible comenzar con una aplicación sencilla en un solo archivo (app.py) para validar rápidamente ideas y luego ir evolucionando hacia una estructura modular usando blueprints y paquetes separados para controladores, servicios y configuración.

Además, Flask se integra de forma natural con otras bibliotecas utilizadas en el proyecto, como PyMongo (para la conexión con MongoDB), Werkzeug (para el manejo de seguridad y hashing de contraseñas) y Jinja2 (para la generación de vistas). Esta integración reduce la fricción técnica y permite centrar los esfuerzos en la lógica específica del dominio: la gestión de ofertas laborales y la generación de estadísticas para la toma de decisiones en la Oficina de RSU.

2.1.4. Bases de Datos NoSQL y MongoDB

El sistema emplea MongoDB como motor de base de datos, el cual pertenece a la categoría de bases de datos NoSQL orientadas a documentos. A diferencia de los sistemas relacionales tradicionales, donde los datos se organizan en tablas con esquemas rígidos, MongoDB almacena la información en documentos con formato similar a JSON, agrupados en colecciones. Cada documento puede tener conjuntos de campos ligeramente diferentes, lo que otorga una gran flexibilidad para manejar datos semi-estructurados.

Esta característica resulta especialmente adecuada para el dominio de las ofertas laborales. Las publicaciones obtenidas mediante web scraping pueden variar en la cantidad y el tipo de información disponible: en algunos casos se incluye el rango salarial, en otros no; ciertos portales detallan el tipo de jornada o el área funcional, mientras que otros solo proporcionan un título y una breve descripción. Con MongoDB, el sistema puede almacenar todos estos casos sin necesidad de redefinir constantemente el esquema de la base de datos.

Desde la perspectiva de rendimiento y análisis, MongoDB ofrece mecanismos de indexación y agregación que permiten realizar búsquedas eficientes y generar estadísticas complejas (por fuente, nivel académico, modalidad, empresa, etc.). En el sistema, estas capacidades se aprovechan mediante la clase MongoDBManager, que encapsula las operaciones sobre la base de datos y expone métodos específicos para las necesidades de la Oficina de RSU (por ejemplo, get_ofertas y get_estadisticas).

2.1.5. Web Scraping como Mecanismo de Obtención de Datos

El web scraping es una técnica que permite recolectar datos de sitios web de forma automatizada. Conceptualmente, implica tres etapas principales: (1) realizar una petición HTTP al sitio de interés para obtener el contenido HTML, (2) analizar la estructura del documento para localizar los elementos que contienen la información relevante y (3) extraer y transformar esos datos en un formato estructurado que pueda ser procesado o almacenado.

En el Sistema de Ofertas Laborales para Tacna, el web scraping es el canal principal de alimentación de datos. Mediante las bibliotecas requests y BeautifulSoup se accede a distintos portales de empleo orientados a la región, se identifica el listado de ofertas en cada página y se extraen atributos como el título del puesto, la empresa, la ciudad, el nivel académico requerido, la modalidad de trabajo y el enlace al detalle de la convocatoria.

Desde un punto de vista teórico, el uso de web scraping debe considerar elementos como:

  • Robustez frente a cambios en el HTML: los portales pueden modificar su diseño, lo que exige que el scraper sea flexible y fácil de actualizar.
  • Respeto a los términos de uso: es necesario revisar las políticas de cada sitio y configurar frecuencias de consulta razonables para no afectar su disponibilidad.
  • Calidad de datos: los datos extraídos deben ser validados, limpiados y normalizados antes de ser incorporados al sistema.

Estos aspectos se abordan en el proyecto mediante funciones de parseo específicas por portal, filtros por ubicación y palabras clave, y procedimientos para evitar la inserción de ofertas duplicadas.

2.1.6. Seguridad, Autenticación y Control de Acceso

Todo sistema que maneja información relevante para la toma de decisiones requiere mecanismos básicos de seguridad y control de acceso. En el caso del Sistema de Ofertas Laborales para Tacna, aunque no se gestionan datos altamente sensibles como historiales clínicos o financieros, sí se trabaja con información que incide en las oportunidades de empleo de los usuarios y en la reputación institucional de la UPT. Por ello, es importante garantizar que solo el personal autorizado pueda administrar el sistema y ejecutar procesos críticos, como la extracción de nuevas ofertas.

Teóricamente, la autenticación se define como el proceso de verificar la identidad de un usuario, mientras que la autorización se refiere a determinar qué acciones puede realizar una vez autenticado. En el sistema, la autenticación se implementa mediante un formulario de inicio de sesión que compara las credenciales ingresadas con los registros almacenados en MongoDB. Las contraseñas no se guardan en texto plano, sino que se protegen mediante funciones de hashing provistas por la librería Werkzeug, reduciendo el riesgo ante accesos no autorizados a la base de datos.

La autorización se materializa a través del decorador login_required, que actúa como una barrera lógica: si una petición se dirige a una ruta protegida y no existe un usuario autenticado en la sesión, el sistema redirige automáticamente al formulario de login y muestra un mensaje de advertencia. De este modo, se evita que usuarios anónimos accedan al dashboard, a las estadísticas o a los procesos de scraping.

2.1.7. Responsabilidad Social Universitaria y Empleabilidad

Desde el ámbito institucional, el proyecto se sustenta en el concepto de Responsabilidad Social Universitaria (RSU), entendido como el compromiso de la universidad con su entorno social y con la formación integral de sus estudiantes. La RSU no se limita a actividades de extensión o voluntariado, sino que abarca la manera en que la institución contribuye al desarrollo humano, económico y cultural de la comunidad en la que está inserta.

Una de las dimensiones centrales de la RSU es la empleabilidad, es decir, la capacidad de los estudiantes y egresados para acceder a un empleo pertinente, digno y acorde con su formación. En este sentido, proveer herramientas que faciliten el acceso a oportunidades laborales constituye una estrategia concreta para materializar la responsabilidad social de la universidad.

El Sistema de Ofertas Laborales para Tacna se inscribe en esta lógica al centralizar y organizar la información sobre ofertas de trabajo disponibles en la región, haciéndola accesible de manera oportuna y amigable para la comunidad universitaria. De esta manera, el sistema no solo es un producto tecnológico, sino también un instrumento que refuerza la misión de la UPT de aportar al desarrollo de Tacna mediante la formación y la inserción laboral de sus profesionales.

2.2. DOCUMENTO SRS (Software Requirements Specification)

El Documento SRS (Software Requirements Specification) es un artefacto fundamental en los procesos de ingeniería de software, ya que recoge de manera estructurada y detallada los requerimientos del sistema. Su propósito es servir como contrato técnico entre los interesados (stakeholders) y el equipo de desarrollo, evitando ambigüedades y malentendidos respecto a lo que el sistema debe hacer.

En el caso del Sistema de Ofertas Laborales para Tacna, el SRS consolida las necesidades identificadas en la Oficina de RSU: automatizar la recolección de ofertas desde portales externos, centralizar la información en una base de datos, permitir la búsqueda y el filtrado según criterios relevantes (empresa, nivel académico, modalidad, palabras clave) y generar estadísticas que apoyen la toma de decisiones. Además, el SRS documenta restricciones y supuestos, como la dependencia de la conectividad a Internet y la disponibilidad del servicio de MongoDB.

2.3. REQUERIMIENTOS FUNCIONALES

Los requerimientos funcionales describen los servicios y comportamientos observables que el sistema debe proporcionar desde la perspectiva del usuario. Responden a la pregunta “¿qué debe hacer el sistema?”. En el Sistema de Ofertas Laborales para Tacna, entre los requerimientos funcionales principales se encuentran:

  • Permitir el inicio y cierre de sesión de usuarios autorizados (personal de RSU).
  • Mostrar un dashboard con las últimas ofertas registradas y estadísticas generales.
  • Listar las ofertas laborales almacenadas aplicando filtros por texto libre, empresa, nivel académico y modalidad.
  • Visualizar el detalle completo de una oferta específica.
  • Ejecutar el proceso de extracción (scraping) desde los portales de empleo configurados.
  • Generar estadísticas agregadas por fuente, nivel académico, modalidad y empresa.

Estos requerimientos funcionales fueron priorizados y refinados en el SRS, sirviendo de base para el diseño de los casos de uso, los diagramas de procesos y el plan de pruebas del sistema.

2.4. REQUERIMIENTOS NO FUNCIONALES

Los requerimientos no funcionales especifican criterios de calidad y restricciones bajo las cuales el sistema debe operar. No describen funcionalidades específicas, sino propiedades tales como rendimiento, seguridad, usabilidad y mantenibilidad. En el contexto del sistema desarrollado, destacan los siguientes:

  • Rendimiento: el sistema debe responder a las peticiones de listado y detalle de ofertas en tiempos aceptables para el usuario (segundos).
  • Disponibilidad: la aplicación debe estar disponible durante el horario laboral de la Oficina de RSU, minimizando tiempos de inactividad.
  • Seguridad: las contraseñas se almacenan mediante hashing y las rutas críticas están protegidas por autenticación.
  • Usabilidad: la interfaz debe ser clara, intuitiva y consistente, dirigida a usuarios no técnicos del ámbito administrativo.
  • Mantenibilidad: la arquitectura por capas y el uso de servicios separados deben facilitar la incorporación de nuevas fuentes de empleo o criterios de filtrado.

Estos requerimientos no funcionales orientan decisiones de diseño (por ejemplo, elección de tecnologías, estructuras de datos y patrones arquitectónicos) y sirven como referencia al evaluar la calidad del sistema desarrollado.

2.5. DOCUMENTO SAD (Software Architecture Document)

El Documento SAD (Software Architecture Document) describe la arquitectura del sistema desde una perspectiva de alto nivel, identificando sus componentes principales, las relaciones entre ellos y las decisiones de diseño más relevantes. Su función es proporcionar una visión global que permita entender cómo está organizado el sistema y por qué se tomaron determinadas decisiones arquitectónicas.

Para el Sistema de Ofertas Laborales para Tacna, el SAD recoge la adopción de una arquitectura por capas con separación entre presentación, lógica de aplicación, servicios de negocio y acceso a datos, así como el uso de Flask como microframework web, MongoDB como base de datos NoSQL y componentes especializados para scraping y estadísticas. También documenta aspectos transversales como el manejo de errores, el registro de eventos (logging) y el tratamiento de la configuración mediante variables de entorno.

2.6. ARQUITECTURA DE SOFTWARE

La arquitectura de software del sistema se concibe como la estructura organizativa que define los componentes, sus responsabilidades y las interacciones entre ellos. Una buena arquitectura facilita el cumplimiento de los requerimientos funcionales y no funcionales, y sienta las bases para la evolución del sistema en el tiempo.

En este proyecto, la arquitectura se articula en torno a tres grandes bloques:

  • Interfaz de usuario (presentación): plantillas HTML, CSS y JavaScript que conforman las vistas de login, dashboard, listado de ofertas, detalle y estadísticas.
  • Lógica de aplicación (controladores): blueprints de Flask que implementan los casos de uso, gestionan las sesiones y coordinan la comunicación con los servicios.
  • Servicios de negocio y datos: clases como MongoDBManager y ScrapingService que encapsulan el acceso a MongoDB y la obtención de datos externos, respectivamente.

Esta organización responde a criterios de modularidad, reutilización y bajo acoplamiento, alineados con las recomendaciones de marcos de trabajo como el Proceso Unificado (RUP) y las buenas prácticas de ingeniería de software.

2.7. DIAGRAMA DE COMPONENTES

El diagrama de componentes es un artefacto UML que representa los módulos lógicos o físicos del sistema y las dependencias entre ellos. En el caso del Sistema de Ofertas Laborales para Tacna, este diagrama identifica componentes como:

  • Componente de autenticación (auth_bp).
  • Componente de dashboard y estadísticas (dashboard_bp).
  • Componente de gestión de ofertas (ofertas_bp).
  • Servicio de base de datos (MongoDBManager).
  • Servicio de scraping (ScrapingService).
  • Componente de configuración (Config).

Representar estos elementos en un diagrama permite visualizar de forma sintética cómo se distribuye la responsabilidad en el sistema y qué componentes se comunican entre sí, facilitando el razonamiento sobre el impacto de cambios futuros.

2.8. DIAGRAMA DE CASOS DE USO

El diagrama de casos de uso es una herramienta de modelado que muestra las interacciones entre los actores externos (usuarios) y el sistema, a través de los servicios que éste ofrece. Para el Sistema de Ofertas Laborales para Tacna, los actores principales son el usuario administrador (personal de RSU) y, potencialmente, otros perfiles que puedan definirse a futuro.

Entre los casos de uso modelados se encuentran: “Iniciar sesión”, “Visualizar dashboard”, “Listar ofertas”, “Filtrar ofertas”, “Ver detalle de oferta”, “Ejecutar extracción de ofertas” y “Consultar estadísticas”. Este diagrama ayuda a asegurar que todos los requerimientos funcionales relevantes se consideren en el diseño y el desarrollo, y proporciona una vista de alto nivel comprensible tanto para usuarios técnicos como no técnicos.

2.9. DIAGRAMA DE PROCESOS Y EDT

El diagrama de procesos describe, mediante notaciones como BPMN o diagramas de actividades, el flujo de trabajo asociado a cada caso de uso o conjunto de casos de uso. En el contexto del sistema, estos diagramas representan, por ejemplo, la secuencia desde que el usuario inicia sesión hasta que visualiza las ofertas actualizadas, o el flujo completo de la ejecución del scraping y la actualización de la base de datos.

Complementariamente, la EDT (Estructura de Desglose de Trabajo) es una herramienta de gestión de proyectos que descompone el trabajo total del proyecto en paquetes manejables. Para el Sistema de Ofertas Laborales para Tacna, la EDT agrupa actividades como análisis de requerimientos, diseño de arquitectura, desarrollo de backend, implementación del servicio de scraping, construcción del frontend, pruebas, despliegue y capacitación. Esta descomposición facilita la planificación, el seguimiento y el control del avance del proyecto.

2.10. METODOLOGÍA DE DESARROLLO INCREMENTAL E ITERATIVA

La metodología de trabajo empleada en el proyecto se basa en un enfoque incremental e iterativo con prototipos evolutivos. En lugar de intentar definir y construir todo el sistema desde el inicio, se optó por desarrollar versiones parciales pero funcionales que fueron refinadas y ampliadas en ciclos sucesivos, incorporando de manera progresiva nuevas funcionalidades y mejoras de calidad.

En una primera iteración se construyó una versión simplificada y monolítica del sistema en app.py, que permitía validar rápidamente el flujo básico de autenticación, visualización de ofertas y navegación entre vistas. A partir de esta base se llevaron a cabo nuevas iteraciones en las que se refactorizó la arquitectura hacia una estructura por capas (controladores, servicios, configuración), se incorporó el servicio de scraping, se enriquecieron los filtros de búsqueda y se añadieron estadísticas de apoyo a la toma de decisiones.

Cada iteración incluyó actividades de análisis (revisión de necesidades y feedback de la Oficina de RSU), diseño (ajustes en la arquitectura y en los modelos de datos), implementación (código de backend, frontend y servicios) y pruebas (verificación de casos de uso clave). Este ciclo repetitivo permitió detectar y corregir problemas de forma temprana, adaptar el alcance del proyecto a los tiempos de las prácticas pre-profesionales y asegurar que el sistema evolucionara alineado con las prioridades reales de la Oficina de Responsabilidad Social Universitaria.

CAPÍTULO IV: APORTE TÉCNICO – ACADÉMICO

4.1. METODOLOGÍA DE DESARROLLO INCREMENTAL E ITERATIVA

El aporte técnico–académico del proyecto se enmarca en la aplicación de una metodología de desarrollo incremental e iterativa, basada en la construcción de prototipos evolutivos. Este enfoque permitió responder de manera flexible a las necesidades de la Oficina de Responsabilidad Social Universitaria (RSU), integrando de forma progresiva nuevos requisitos y mejoras arquitectónicas sin perder de vista la viabilidad del proyecto en el contexto de las prácticas pre-profesionales.

4.1.1. Inicio del Proceso de Desarrollo

4.1.1.1. Problemática

La problemática inicial identificada en la Oficina de RSU se relacionaba con la gestión manual y dispersa de las ofertas laborales. El personal debía visitar diversos portales de empleo, filtrar visualmente las oportunidades relevantes para Tacna, copiar la información a hojas de cálculo y luego difundirla por canales internos. Este proceso consumía entre 2 y 3 horas diarias, era propenso a errores, dificultaba el seguimiento histórico de las ofertas y no permitía obtener estadísticas confiables sobre el mercado laboral regional.

4.1.1.2. Formulación del Problema

A partir de la situación descrita, el problema se puede formular de la siguiente manera:

¿Cómo diseñar e implementar un sistema de información que permita automatizar la recolección, el almacenamiento, la búsqueda y el análisis de ofertas laborales para la región de Tacna, reduciendo el tiempo de trabajo manual de la Oficina de RSU y mejorando la calidad y oportunidad de la información disponible para la comunidad universitaria de la UPT?

4.1.1.3. Justificación e Importancia

La justificación del proyecto se sustenta en tres dimensiones principales:

  • Operativa: al automatizar la búsqueda y el registro de ofertas, se reduce el esfuerzo manual del personal de RSU y se liberan horas que pueden destinarse a actividades de mayor valor añadido, como el acompañamiento personalizado a estudiantes y egresados.
  • Estratégica: contar con un sistema que centralice y organice las ofertas laborales fortalece la capacidad de la universidad para articularse con el entorno productivo y posicionarse como un actor relevante en la empleabilidad regional.
  • Académica y social: la herramienta contribuye directa y concretamente a la Responsabilidad Social Universitaria, al facilitar el acceso a oportunidades laborales y apoyar la inserción de los profesionales formados por la UPT en el mercado de trabajo.

4.1.1.4. Objetivos

En coherencia con la problemática y la justificación, se establecieron los siguientes objetivos:

Objetivo general

Diseñar e implementar un Sistema de Ofertas Laborales para Tacna, basado en una metodología de desarrollo incremental e iterativa, que automatice la recolección, la gestión y el análisis de ofertas laborales para la comunidad universitaria de la UPT.

Objetivos específicos

  • Analizar los procesos actuales de la Oficina de RSU relacionados con la gestión de ofertas laborales.
  • Definir y documentar los requerimientos funcionales y no funcionales del sistema.
  • Diseñar una arquitectura por capas que separe presentación, lógica de aplicación y servicios de datos.
  • Implementar un módulo de scraping que extraiga ofertas desde portales de empleo relevantes para Tacna.
  • Desarrollar una interfaz web que permita buscar, filtrar y visualizar ofertas y estadísticas asociadas.
  • Validar el sistema mediante pruebas y obtener retroalimentación del personal de la Oficina de RSU.

4.1.1.5. Solución Propuesta

La solución propuesta consiste en una aplicación web construida sobre el microframework Flask, utilizando MongoDB como base de datos NoSQL para almacenar las ofertas laborales y los usuarios, y un servicio de web scraping basado en requests y BeautifulSoup para la recolección automática de oportunidades desde distintos portales de empleo. La aplicación se estructura en capas y componentes bien definidos, lo que facilita su mantenimiento y ampliación.

Desde el punto de vista metodológico, la solución se fue construyendo en incrementos: primero una versión mínima funcional centrada en el login y el listado de ofertas, luego la incorporación del scraping y finalmente la adición de estadísticas y mejoras en la experiencia de usuario. Cada incremento fue evaluado junto con la supervisora de la Oficina de RSU, lo que aseguró la alineación constante con las necesidades institucionales.

4.1.1.6. Factibilidad

La factibilidad del proyecto se analizó en tres dimensiones:

  • Factibilidad técnica: las tecnologías seleccionadas (Python, Flask, MongoDB, requests, BeautifulSoup) son de uso extendido, cuentan con abundante documentación y pueden ejecutarse en la infraestructura disponible en la universidad (equipos de desarrollo y servidores locales o en la nube).
  • Factibilidad económica: todas las herramientas empleadas son de código abierto o disponen de versiones gratuitas suficientemente potentes para las necesidades del proyecto, por lo que no se requieren inversiones adicionales en licencias de software.
  • Factibilidad operativa: el personal de la Oficina de RSU puede utilizar el sistema mediante una interfaz web sencilla, y la capacitación requerida es moderada. Además, el mantenimiento básico (ejecución del scraping, revisión de ofertas, actualización de parámetros) puede ser asumido por el equipo de RSU con el apoyo puntual de personal técnico de la universidad.

4.1.2. Iteraciones de Diseño y Arquitectura

4.1.2.1. Elaboración del Documento SRS

En la primera serie de iteraciones se elaboró el Documento SRS, que recoge de manera estructurada los requerimientos funcionales y no funcionales del sistema. Este documento se construyó progresivamente: partiendo de una versión inicial basada en entrevistas con la supervisora de RSU y refinándola a medida que se iban validando prototipos y descubriendo nuevas necesidades o restricciones.

El enfoque incremental permitió que el SRS no fuera un artefacto estático, sino un documento vivo que se fue ajustando conforme se obtenía experiencia con el sistema, manteniendo siempre una traza clara entre los requisitos establecidos y las funcionalidades implementadas.

4.1.2.2. Elaboración del Documento SAD

Paralelamente, se elaboró el Documento SAD, que describe la arquitectura de software adoptada. En las primeras iteraciones, el SAD se centró en documentar la versión monolítica inicial y las decisiones relacionadas con la selección de tecnologías. En iteraciones posteriores, se fue enriqueciendo con la descripción de la arquitectura por capas, los componentes (controladores, servicios, utilidades) y los patrones utilizados (MVC ligero, separación de responsabilidades).

De esta forma, el SAD refleja la evolución arquitectónica del sistema y sirve como referencia para comprender cómo los distintos módulos colaboran para cumplir los objetivos definidos en el SRS.

4.1.3. Construcción Incremental del Sistema

La fase de construcción no se abordó como un bloque único, sino como una sucesión de incrementos funcionales. Cada incremento tenía un objetivo claro (por ejemplo, “implementar autenticación básica”, “agregar scraping de un portal”, “incorporar estadísticas por fuente”) y culminaba con una versión ejecutable del sistema que podía ser demostrada y validada.

Este modo de trabajo permitió integrar tempranamente los distintos componentes (backend, scraping, base de datos y frontend), identificar problemas de integración y usabilidad y corregirlos antes de avanzar hacia funcionalidades más complejas. Desde la perspectiva académica, la construcción incremental reforzó la comprensión de la relación entre diseño, implementación y pruebas, y evidenció la importancia de recibir retroalimentación continua de los usuarios finales para lograr un sistema útil y sostenible.
