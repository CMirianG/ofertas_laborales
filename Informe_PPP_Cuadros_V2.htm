INFORME DE PRÁCTICAS PRE PROFESIONALES

Sistema de Ofertas Laborales para Tacna

INTRODUCCIÓN

El presente informe describe la experiencia de Prácticas Pre Profesionales desarrollada en la Oficina de Responsabilidad Social Universitaria (RSU) de la Universidad Privada de Tacna (UPT), cuyo principal producto fue el diseño e implementación del Sistema de Ofertas Laborales para Tacna.

Este sistema web tiene como propósito centralizar, automatizar y facilitar la consulta de ofertas laborales dirigidas a estudiantes y egresados de la UPT, utilizando tecnologías modernas como Flask, MongoDB y requests + BeautifulSoup para web scraping. De esta manera, se reduce el trabajo manual que realizaba la Oficina de RSU al recopilar oportunidades de empleo en diversos portales, pasando de procesos de 2–3 horas diarias a tareas ejecutadas en pocos minutos.

El informe se estructura en cuatro capítulos: el primero presenta las generalidades del centro de prácticas; el segundo desarrolla el marco teórico y la arquitectura del sistema; el tercero detalla las actividades realizadas durante el período de prácticas; y el cuarto expone el aporte técnico-académico, los desafíos afrontados y el impacto en la formación profesional. Finalmente, se incluyen conclusiones y recomendaciones orientadas a la mejora continua del sistema y de los procesos institucionales.

CAPÍTULO II: MARCO TEÓRICO Y ARQUITECTURA DEL SISTEMA

2.1. FUNDAMENTOS CONCEPTUALES

El Sistema de Ofertas Laborales para Tacna se apoya en varios conceptos clave de la ingeniería de software y de los sistemas de información:

  • Aplicaciones web basadas en microframeworks: uso de Flask para construir aplicaciones ligeras, modulares y extensibles.
  • Bases de datos NoSQL: uso de MongoDB para almacenar documentos flexibles, adecuados para datos semi-estructurados como ofertas laborales.
  • Web scraping: técnica para extraer información estructurada desde páginas web utilizando librerías como requests y BeautifulSoup.
  • Patrones MVC y arquitectura por capas: separación de responsabilidades en capas de presentación, lógica de negocio y acceso a datos.
  • Seguridad y autenticación: uso de sesiones, hash de contraseñas y control de acceso para proteger la información del sistema.

2.2. ARQUITECTURA TÉCNICA

2.2.1. Stack Tecnológico Principal

Capa Tecnología Descripción
Backend web Flask (Python) Microframework para definir rutas, controladores, sesiones y manejo de peticiones HTTP.
Base de datos MongoDB + PyMongo Almacenamiento de ofertas laborales y usuarios en documentos flexibles.
Scraping requests, BeautifulSoup Extracción y parseo de ofertas desde múltiples portales web de empleo.
Configuración python-dotenv, clase Config Manejo de variables de entorno y parámetros del sistema (URIs, límites, timeouts, etc.).
Frontend HTML5, CSS, JS, plantillas Jinja2 Interfaz para login, dashboard, listados y visualización de estadísticas.

2.2.2. Patrones Arquitectónicos Implementados

La solución adopta un enfoque de MVC ligero sobre Flask:

  • Modelo / Servicios: encapsulados en MongoDBManager y ScrapingService, que gestionan acceso a datos y lógica de negocio.
  • Controladores: agrupados en auth.py, dashboard.py y ofertas.py como blueprints, responsables del flujo de cada caso de uso.
  • Vistas: plantillas HTML en templates/, conectadas a los controladores mediante renderización Jinja2.

Además, se utiliza una arquitectura por capas:

  • Capa de presentación (UI): plantillas y archivos estáticos.
  • Capa de aplicación (controladores): orquesta la lógica entre vistas y servicios.
  • Capa de servicios (negocio y datos): scraping y acceso a MongoDB.
  • Capa de configuración/utilidades: clase Config y módulos de utils.

2.3. DISEÑO DE COMPONENTES

2.3.1. Componente de Autenticación y Autorización

El componente de autenticación se implementa mediante un blueprint de Flask (auth_bp), que gestiona el inicio y cierre de sesión, así como la validación de credenciales utilizando contraseñas almacenadas mediante hash. Se utilizan sesiones de Flask para mantener el estado del usuario y un decorador login_required para proteger las rutas críticas como el dashboard y la gestión de ofertas.

Este componente incluye un modo offline: cuando la base de datos no está disponible, se habilita un usuario de emergencia (admin) que permite continuar utilizando el sistema con capacidades limitadas, lo que incrementa la resiliencia frente a fallos de infraestructura.

2.3.2. Componente de Extracción de Datos (Scraping)

El servicio de scraping, encapsulado en ScrapingService, concentra la lógica para conectarse a diferentes portales de empleo, descargar el HTML de las páginas, parsearlo con BeautifulSoup y transformar los resultados en estructuras normalizadas.

El componente está diseñado para:

  • Gestionar múltiples fuentes de datos mediante URLs configurables en la clase Config.PORTALS.
  • Aplicar filtros geográficos y semánticos (por ejemplo, ofertas en Tacna).
  • Evitar duplicados y actualizar ofertas existentes en la base de datos.

2.3.3. Componente de Gestión de Datos

El componente de gestión de datos está representado por MongoDBManager, responsable de:

  • Establecer y verificar la conexión a MongoDB.
  • Proveer métodos de alto nivel como get_ofertas, get_oferta_by_id y get_estadisticas.
  • Implementar agregaciones para estadísticas por fuente, nivel académico, modalidad y empresa.

Este diseño reduce el acoplamiento entre controladores y base de datos, facilitando futuras migraciones o cambios de tecnología de persistencia si fueran necesarios.

2.5.2. Principios Guía del Desarrollo

  • Simplicidad primero: priorizar una versión funcional sencilla antes de optimizar o añadir complejidad.
  • Bajo acoplamiento y alta cohesión: cada módulo tiene una responsabilidad clara y bien delimitada.
  • Reutilización: componentes como MongoDBManager y ScrapingService son reutilizables para futuros proyectos.
  • Orientación a pruebas manuales frecuentes: validación continua del scraping y las rutas web para evitar fallos silenciosos.

CAPÍTULO III: ACTIVIDADES DESARROLLADAS

3.1. ACTIVIDADES DE ANÁLISIS Y DISEÑO

En la fase inicial se analizaron los procesos de la Oficina de RSU relacionados con la búsqueda de ofertas laborales: fuentes de información, criterios de filtrado, frecuencia de actualización y formato de difusión a estudiantes. A partir de este análisis se elaboraron diagramas de flujo y un modelo de datos orientado a MongoDB.

Se diseñaron las vistas principales del sistema (login, dashboard, listado de ofertas, detalle de oferta y estadísticas), así como la interacción entre los controladores y los servicios de base de datos y scraping.

3.2. ACTIVIDADES DE DESARROLLO TÉCNICO

3.2.1. Configuración del Entorno de Desarrollo

Se configuró el entorno de desarrollo instalando Python, MongoDB y las dependencias listadas en requirements.txt. Se utilizaron entornos virtuales para aislar las librerías del proyecto y se empleó un repositorio Git para el control de versiones.

3.2.2. Implementación del Backend

El backend se implementó inicialmente en un archivo app.py con una versión monolítica, y posteriormente se modularizó en controladores y servicios. Se definieron rutas para:

  • Autenticación (login, logout).
  • Dashboard con últimas ofertas y estadísticas.
  • Listado paginado y filtrado de ofertas.
  • Visualización detallada de una oferta específica.
  • API interna para solicitudes AJAX y extracción de ofertas.

3.2.3. Desarrollo del Servicio de Scraping

Se desarrolló el servicio de scraping para conectarse a los portales definidos en Config.PORTALS, extrayendo información relevante como título, empresa, ubicación, nivel académico requerido y modalidad de trabajo. El servicio transforma la información HTML en documentos listos para ser almacenados en MongoDB.

3.2.4. Implementación del Frontend

Se implementaron las plantillas HTML con una interfaz limpia y centrada en la usabilidad: menú de navegación, formularios de login, tablas de ofertas con filtros y una sección de estadísticas gráficas. Se aplicó CSS personalizado y scripts JavaScript ligeros para manejar ciertas interacciones (por ejemplo, llamadas AJAX para la extracción de ofertas).

3.3. ACTIVIDADES DE INTEGRACIÓN Y PRUEBAS

3.3.1. Pruebas de Componentes

Se realizaron pruebas unitarias manuales sobre funciones clave del scraping y de acceso a datos, verificando que cada componente cumpliera su responsabilidad de forma aislada: conexión a MongoDB, parseo de HTML y formateo de datos para la vista.

3.3.2. Pruebas de Integración

Posteriormente se validó el flujo completo desde el momento en que se lanza la extracción hasta la visualización de las nuevas ofertas en la interfaz. Se prestó especial atención a los casos de error, como la indisponibilidad temporal de un portal o de la base de datos.

3.3.3. Pruebas de Rendimiento

Aunque el sistema está orientado a un volumen moderado de datos, se realizaron pruebas de rendimiento básicas, observando el tiempo de respuesta de las páginas y el tiempo de ejecución del proceso de scraping con múltiples portales, para garantizar una experiencia de usuario fluida.

3.4. ACTIVIDADES DE IMPLEMENTACIÓN Y CAPACITACIÓN

3.4.1. Despliegue en Entorno de Producción

Se configuró un entorno de ejecución para el sistema, definiendo variables de entorno, el URI de MongoDB y parámetros de seguridad como la clave secreta de Flask. Se documentaron los pasos de despliegue para facilitar futuras actualizaciones.

3.4.2. Capacitación al Personal

Finalmente, se realizó una sesión de capacitación dirigida al personal de la Oficina de RSU, explicando el uso del sistema: ingreso al dashboard, filtros de búsqueda, interpretación de estadísticas y ejecución del proceso de extracción de nuevas ofertas.

CAPÍTULO IV: APORTE TÉCNICO-ACADÉMICO

4.1. INTEGRACIÓN DE CONOCIMIENTOS ACADÉMICOS

4.1.1. Aplicación de Principios de Ingeniería de Software

El proyecto permitió aplicar de manera directa principios de la ingeniería de software, tales como modularidad, reutilización, separación de responsabilidades, diseño por capas y manejo de configuración. Asimismo, se emplearon buenas prácticas de control de versiones y documentación.

4.1.2. Desarrollo de Competencias Técnicas Especializadas

Se consolidaron competencias en desarrollo web con Python y Flask, manejo de bases de datos NoSQL, técnicas de scraping y procesamiento de datos semi-estructurados. Estas competencias son altamente demandadas en el mercado laboral actual.

4.2. INNOVACIÓN TÉCNICA Y ADAPTACIÓN AL CONTEXTO

4.2.1. Diseño de una Arquitectura Híbrida

El sistema combina una versión monolítica simplificada con una arquitectura modular basada en blueprints y servicios, lo que facilita tanto la comprensión inicial del proyecto como su evolución futura hacia un sistema más escalable y mantenible.

4.2.2. Sistema de Configuración Dinámica

La clase Config centraliza parámetros clave (URIs de portales, límites de paginación, timeouts, etc.), permitiendo adaptar el comportamiento del sistema sin modificar el código fuente. Esto habilita la incorporación de nuevos portales y el ajuste de parámetros operativos de forma ágil.

4.3. DESAFÍOS TÉCNICOS SUPERADOS

4.3.1. Heterogeneidad de Fuentes de Datos

Cada portal de empleo presenta estructuras HTML distintas y criterios de búsqueda propios. El reto consistió en diseñar un scraping lo suficientemente flexible para adaptarse a estas diferencias, manteniendo un modelo de datos unificado en MongoDB.

4.3.2. Manejo de Datos Semi-estructurados

Las ofertas laborales no siempre contienen todos los campos de forma uniforme. Fue necesario diseñar estrategias para manejar valores faltantes, normalizar textos y garantizar que la información se presentara de forma clara al usuario final.

4.3.3. Balance entre Automatización y Calidad

Automatizar la extracción masiva de ofertas implica el riesgo de incluir información irrelevante o duplicada. Se implementaron filtros por ubicación (Tacna), palabras clave y verificaciones de duplicados para mantener un equilibrio entre volumen y calidad de las ofertas registradas.

4.4. IMPACTO EN LA FORMACIÓN PROFESIONAL

4.4.1. Desarrollo de Competencias de Gestión de Proyectos

La planificación de fases, la estimación de tiempos y la priorización de tareas fortalecieron las competencias en gestión de proyectos, alineadas con las buenas prácticas de la disciplina.

4.4.2. Comprensión de la Dimensión Social de la Tecnología

El proyecto evidenció el rol de la tecnología como herramienta para mejorar la empleabilidad y la vinculación universidad-sociedad, reforzando la comprensión de la dimensión social de los sistemas de información.

4.4.3. Integración de Ética y Responsabilidad Profesional

El manejo responsable de los datos, la transparencia en la forma en que se presentan las ofertas y la consideración de la accesibilidad para los usuarios son aspectos éticos que se integraron en el diseño y desarrollo del sistema.

CONCLUSIONES

El Sistema de Ofertas Laborales para Tacna constituye una solución tecnológica innovadora que optimiza significativamente los procesos de la Oficina de Responsabilidad Social Universitaria de la UPT. La arquitectura implementada demuestra que es posible desarrollar un sistema robusto y eficiente utilizando tecnologías modernas como Flask, MongoDB y BeautifulSoup.

Gracias a la automatización del proceso de búsqueda y registro de ofertas, se reduce de forma notable el tiempo de dedicación manual, se centraliza la información en una única base de datos y se ofrecen herramientas avanzadas de búsqueda y filtrado para la comunidad universitaria.

La experiencia de desarrollo permitió aplicar conocimientos académicos en un contexto real, demostrando la viabilidad técnica, económica y operativa del proyecto, y estableciendo una base sólida para futuras expansiones y mejoras del sistema.
